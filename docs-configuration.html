YAML Configuration

Configuration Structure

DIME configurations are organized into three main sections:

# main.yaml - Primary configuration file
app:
  license: YOUR-LICENSE-KEY
  ring_buffer: 4096
  http_server_uri: http://127.0.0.1:9999/
  ws_server_uri: ws://127.0.0.1:9998/

sources:
  - name: plc_data
    connector: ModbusTCP
    # Source configuration...

sinks:
  - name: database
    connector: PostgreSQL
    # Sink configuration...

Modular Configuration

Split configurations across multiple files using YAML anchors for better organization:

# plc.yaml - PLC connector definitions
plc1: &plc1
  name: production_plc
  connector: EthernetIP
  address: 192.168.1.100
  items:
    - name: temperature
      address: Temperature_Tag
      type: float
    - name: pressure
      address: Pressure_Tag
      type: float

# database.yaml - Database sink definition
postgres: &postgres
  name: timeseries_db
  connector: PostgreSQL
  connection_string: "Host=localhost;Database=dime;Username=user;Password=pass"
  table_name: sensor_data

# main.yaml - Reference the modular configurations
sources:
  - *plc1

sinks:
  - *postgres

Configuring Data Items

Define individual data points with addresses, types, and transformation scripts:

items:
  - name: temperature_celsius
    address: 40001
    type: float
    rbe: true  # Report by exception
    every: 5   # Read every 5 scan intervals
    script: |
      -- Convert Fahrenheit to Celsius
      return (result - 32) * 5/9
    sink:
      mtconnect: Device[name=machine1]/Controller/Temperature[category=Sample]

Data Transformation with Scripting

Use Lua or Python scripts for complex data transformations:

- name: opc_source
  connector: OpcUa
  lang_script: Lua  # or Python
  init_script: |
    -- Initialize libraries and variables
    json = require('json')
    alarm_threshold = 100
    
  item_script: |
    -- Process each item
    if result > alarm_threshold then
      emit('./alarm', 'HIGH_TEMPERATURE')
    end
    return result * 1.5  -- Scale value
    
  items:
    - name: sensor1
      node_id: ns=2;s=Sensor1
      script: |
        -- Item-specific processing
        local scaled = result / 100
        return string.format("%.2f", scaled)

Common Scripting Patterns

Data Aggregation

-- Calculate average of multiple sensors
local sum = 0
local count = 0
for i = 1, 5 do
  local val = cache(string.format("sensor%d/value", i), 0)
  sum = sum + val
  count = count + 1
end
return sum / count

Threshold Alerting

-- Generate alerts based on thresholds
if result > 100 then
  emit("./alert", "HIGH")
  emit("./alert_message", "Temperature exceeds limit: " .. result)
elseif result < 20 then
  emit("./alert", "LOW")
  emit("./alert_message", "Temperature below minimum: " .. result)
else
  emit("./alert", "NORMAL")
end
return result

Data Transformation

-- Parse complex data structures
local json = require('json')
local data = json.decode(result)

-- Extract specific fields
emit("./temperature", data.sensors.temperature)
emit("./pressure", data.sensors.pressure)
emit("./status", data.machine.status)

-- Calculate derived values
local efficiency = (data.output / data.input) * 100
emit("./efficiency", efficiency)

return nil  -- Don't emit original message

Configuration Best Practices

üèóÔ∏è Structure & Organization

  • Use separate YAML files for different connector types
  • Group related connectors together
  • Use meaningful, descriptive names
  • Document complex configurations with comments

üîí Security

  • Store credentials in environment variables
  • Use encrypted connections when available
  • Implement least-privilege access
  • Regular security audits of configurations

‚ö° Performance

  • Optimize scan intervals for data requirements
  • Use Report By Exception (RBE) to reduce data volume
  • Batch database writes for efficiency
  • Monitor connector execution times

üîß Maintenance

  • Version control all configuration files
  • Test configurations in staging environment
  • Document custom scripts and transformations
  • Regular backup of configurations
